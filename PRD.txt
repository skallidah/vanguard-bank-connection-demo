SECTION 1: APPLICATION OVERVIEW
--------------------------------------------------------------------------------

The Vanguard Bank Connection Demo is an internal prototype of the bank account
linking flow used on vanguard.com. It allows a Vanguard customer to:

  1. Enter external bank account details (routing number, account number, type,
     owner name).
  2. Review which Vanguard investment accounts will be authorized to transact
     with the linked bank.
  3. Confirm a summary and submit, which triggers micro-deposit verification.

The app is a single Flask server exposing both a multi-page web UI and a REST
API. It uses an Excel file (data.xlsx) as its database, seeded on first run
from mock-db.json.

SECTION 3: PAGES & ROUTES
--------------------------------------------------------------------------------

PAGE 1 — Landing / Intro
  Route:    GET /
  Title:    "Connect your bank account"
  Purpose:  Explains the 4-step micro-deposit process to the user.
  Action:   Single CTA button labeled "CONTINUE" links to /add-bank.

PAGE 2 — Add Bank
  Route:    GET /add-bank
  Title:    "Add a new bank"
  Purpose:  Collect bank and owner details. Validates on submit client-side,
            then calls POST /bff/bank-connections/draft.
  Fields:
    - Routing number (required, exactly 9 digits)
      On blur with a valid 9-digit value: auto-fetches bank name via
      GET /routing-lookup?routingNumber=<value> and populates:
        • Bank name (read-only display)
        • Correspondent routing number (read-only input, if applicable)
        • Correspondent account number (read-only input, if applicable)
    - Account number (required)
    - Re-enter account number (required, must match account number)
    - Account type: radio — Checking (default) | Savings
    - Owner first name (required)
    - Owner last name (required)
    - Owner middle name (optional)
    - Owner suffix (optional dropdown)
    - Bank nickname (optional text input)
    - Make this your primary bank (optional checkbox, default checked)
  On success: redirects to /authorize/<draftId>

PAGE 3 — Authorize Accounts
  Route:    GET /authorize/<draft_id>
  Title:    "Authorize accounts"
  Purpose:  Shows bank details from the draft and lists which Vanguard
            accounts are auto-authorized (identical ownership) vs. eligible
            for additional authorization (partial ownership match).
  Sections:
    - Bank information table (bank name, routing number, account number,
      account type, account owners, correspondent details if present)
    - "Accounts that will be authorized" — auto-authorized list
    - "Review additional accounts" — eligible additional accounts list
  Action:   "REVIEW" button calls POST /bff/bank-connections/draft/<id>/authorizations
            with the auto-authorized account IDs, then redirects to /review/<draftId>
  Other:    CANCEL → /, BACK → /add-bank

PAGE 4 — Review
  Route:    GET /review/<draft_id>
  Title:    "Review your information"
  Purpose:  Final summary before submission.
  Sections:
    - Info box with next-step instructions (micro-deposit process)
    - Bank information: bank name, routing number, account type, account
      owners, nickname (if set), primary bank (Yes/No), verification
      method (always "MICRO_DEPOSIT")
    - Authorized accounts list (name — account name — ID, with "(Managed)"
      badge if applicable)
  Action:   "SUBMIT" button calls POST /bff/bank-connections/draft/<id>/submit,
            then redirects to /success/<bankConnectionId>
  Other:    CANCEL → /, BACK → /authorize/<draftId>, Edit → /add-bank

PAGE 5 — Success
  Route:    GET /success/<bc_id>
  Title:    "Submission Successful"
  Purpose:  Confirms the bank was linked and micro-deposits initiated.
  Content:
    - Green checkmark icon
    - "Entry submitted successfully"
    - "Your bank account has been connected and micro-deposit verification
       has been initiated."
    - "It may take 1–3 business days for the deposits to appear."
    - Bank Connection ID displayed in monospace (e.g. bc_xxxxxx)
  Action:   "START OVER" button → /

SECTION 7: API ENDPOINTS SUMMARY (for direct API test cases)
--------------------------------------------------------------------------------

GET  /routing-lookup?routingNumber=<9digits>
  200: { bankName, correspondentRoutingNumber, correspondentAccountNumber }
  404: { bankName: null } — unknown routing number

POST /bff/bank-connections/draft
  Body: { routingNumber, accountNumber, accountType, owners:[{firstName,lastName}],
          nickname?, makePrimary?, correspondentRoutingNumber?, correspondentAccountNumber? }
  201: { draftId, status:"DRAFT" }

PUT  /bff/bank-connections/draft/<draftId>
  Body: any subset of draft fields
  200: { draftId, status:"DRAFT" }

GET  /bff/bank-connections/draft/<draftId>/authorize
  200: { bankInformation:{...}, autoAuthorizedAccounts:[...], eligibleAdditionalAccounts:[...] }
  404: { error:"Draft not found" }

POST /bff/bank-connections/draft/<draftId>/authorizations
  Body: { authorizedVanguardAccounts: [<vaId>, ...] }
  200: { draftId, authorizedVanguardAccounts:[...] }

GET  /bff/bank-connections/draft/<draftId>/review
  200: { bankInformation:{...}, authorizedAccounts:[...], primaryBank, verificationMethod }
  404: { error:"Draft not found" }

POST /bff/bank-connections/draft/<draftId>/submit
  201: { bankConnectionId, status:"PENDING_VERIFICATION" }
  404: { error:"Draft not found" }

--------------------------------------------------------------------------------
SECTION 8: NOTES FOR KANEAI TEST GENERATION
--------------------------------------------------------------------------------

1. All browser-based tests should use the LT Tunnel URL as the base URL.

2. The database is file-based (data.xlsx). To keep tests independent, prefer
   using the pre-seeded draft IDs (draft_001 through draft_005) for read-only
   steps. For end-to-end submission tests, delete data.xlsx before each run
   to reset to seed state.

3. The app has no authentication layer. All routes are publicly accessible.

4. Client-side validation fires on the CONTINUE button click. All validation
   errors listed in Section 6 are rendered in the DOM without a page reload.

5. After clicking REVIEW on the authorize page, the redirect to /review is
   triggered by JavaScript after a successful API response. Allow for async
   navigation in test assertions.

6. After clicking SUBMIT on the review page, the redirect to /success/<bc_id>
   is similarly JavaScript-driven. The bc_id in the URL is the newly created
   bank connection ID (format: bc_ followed by 6 hex characters).

7. The "API Calls" log panel at the bottom of add-bank and authorize pages
   can be used in tests as a secondary assertion for which API calls were made.
